## DECISION SUMMARY

The `/place-order` catalog grid cell semantics flipped from BOXES to UNITS. Typing `12` in a 50ML cell now means "12 units of 50ML" (= 1 carton), not "12 cartons" as it would have meant before. To preserve the carton-ordering speed, `+` and `=` keys add one box worth of units (`qty + boxSize`), and `-` and `_` subtract one box (floor 0). Hover/focus reveals subtle 16×14px +/− buttons on the right edge of each cell that mirror the keyboard. The mailto body to depot stays byte-identical to pre-flip: cart now stores units, email builder is a no-op pass-through, depot reads `50ML*12` exactly as it always did. Cart panel chips display `×N · M box` when units is a clean multiple of step (e.g. `×12 · 1 box`), or `×N` alone otherwise — including the step=1 suppression for drums and bags. Column headers gained a `packContainerLabel()` helper decoupled from `packStep()`: cartonable packs continue to show "box 12 / box 6 / box 4", drums (10L / 20L / 30L) show "drum", 40KG shows "bag", and unknown packs show no suffix. The 10L `packStep` value was corrected from 2 to 1 in the same change — it's a drum at this depot, not a box of 2.

The session opened as a behaviour change with the audit and 11-phase plan running cleanly, but two material bugs were caught at audit by Claude Code that the original session brief had missed: the cart panel volume total formula `qty × packStep × packToLitres` would have ballooned 6-12× under the new semantics (10.2 L → 122.4 L), and the last-order-recall pathway had a `unitsToBoxes` division that would have silently turned recalled quantities into 1/12th of themselves (24 units becoming a 2-unit recall). Both were folded into the implementation phases. Five smaller in-session corrections also landed: the "N box" hint was pulled out of the cell entirely after a visual review (cells became too crowded), the +/− buttons were moved from left edge to right edge with bigger glyphs after the first eyeball test, the Tailwind `group-focus-within` syntax was swapped for the arbitrary-variant form to handle older Tailwind versions, the bottom hint bar chip was added to three separate render paths (not just `PaginationFooter` as initially planned), and the 10L packStep data fix was discovered when the column header "10L · box 1" reading was flagged.

## CONTEXT CHANGES

- **Cart `packQtys` semantics flipped from BOXES to UNITS.** `CartLine.packQtys[pack]` now stores the unit count the operator wants to order, not the carton count. `types.ts` JSDoc rewritten to reflect this and reference the May 12 supersession of the prior boxes-semantics memory note.
- **Email builder is a no-op pass-through.** `lib/place-order/email.ts:75-80` removed the `× packStep(label)` multiplication. The emitted line `${label}*${units}` now uses `line.packQtys[k]` directly. File-top comment block and `EmailLine.packQtys` JSDoc rewritten to document the new semantics. The `packStep` import was dropped from this file (now unused).
- **Cell input behaviour.** Typing 0-9 writes units directly to `packQtys` — no multiplier. `+` and `=` keys call `onSetQty(qty + boxSize)`. `-` and `_` keys call `onSetQty(Math.max(0, qty - boxSize))`. All four use `e.preventDefault()`. Native key-repeat handles "hold to repeat." Existing arrow/digit/Esc/PageUp handlers preserved.
- **+/− buttons in cells.** Two absolute-positioned `<button>` elements inside the wrapper div. Width 16px, height 14px. Position: `+` at top-right (`right-[1px] top-[1px]`), `−` at bottom-right (`right-[1px] bottom-[1px]`). Style: `text-gray-400 hover:text-gray-700 hover:bg-gray-100 rounded-[2px] text-[11px] leading-none`, neutral grey only — no teal. Visibility: `opacity-0` by default, `opacity-100` on `group-hover` OR `[.group:focus-within_&]` (the arbitrary-variant form, which compiles to plain CSS and works regardless of Tailwind version). `tabIndex={-1}` on both buttons (out of Tab cycle). `onMouseDown={(e) => e.preventDefault()}` on both — critical for keeping focus on the input when the button is clicked.
- **Cart panel pack chip format.** `cart-panel.tsx` chip render now computes `const step = packStep(formatPack(pack))` and `const isClean = step > 1 && units > 0 && units % step === 0`. Primary span: `×{units}` in `text-gray-700 font-mono font-semibold`. Conditional secondary span: `· {units / step} box` in `text-gray-400 font-normal ml-1` when `isClean`. Output: `×12 · 1 box` for clean multiples; `×13` alone for non-clean; `×5` alone for step=1 drums.
- **Cart panel volume total formula.** `cart-panel.tsx:63-75` removed the `× packStep(formatPack(pack))` factor. New shape: `sum += units * packToLitres(pack)`. This was the bug trap — without this fix, totals would have multiplied a second time (12 units of 50ML showing as 7.2 L instead of 0.6 L).
- **Column header `packContainerLabel()` helper.** New exported function in `lib/place-order/pack.ts`. Returns `"box 12" | "box 6" | "box 4" | "drum" | "bag" | null`. Decoupled from `PACK_STEP_MAP` (math) by design — a future pack could have step=1 without being a drum (hypothetical loose can). `variant-grid.tsx` header render now uses this helper. When the helper returns null, the header renders just the pack label with no separator dot, no suffix.
- **`PACK_STEP_MAP[10L] = 1`** (was 2). 10L is a drum at this depot, not a box of 2. Discovered during column header verification, fixed in `pack.ts`.
- **Last-order-recall pathway.** `last-order-recall.tsx` `unitsToBoxes` helper deleted entirely. `RepeatOrderEntry.boxes` renamed to `.units`. `e.boxes > 0` filter became `e.units > 0`. `itemText` template became `${baseDisp} ${packLabel} ×${units}`. `place-order-page.tsx` `handleRepeatOrder` updated to reference `entry.units`. Per locked Q2 decision: lines that previously rounded to 0 boxes (e.g. parser-emitted 7 units of 50ML where step=12) now appear in the recall list as `×7` instead of being silently dropped. `RecallLine.isCarton` / `cartonCount` fields preserved in the type but no longer read — Parser v6.5 always populates `quantity` correctly.
- **Draft persistence.** `STORAGE_KEY` bumped from `"orbitoms_place_order_draft_v1"` to `"orbitoms_place_order_draft_v2"`. Module-top one-shot `localStorage.removeItem("orbitoms_place_order_draft_v1")` wrapped in `typeof window !== "undefined"` for Next.js SSR safety. Operators with an in-flight pre-flip draft would see an empty cart on next load — accepted risk per locked decision (no drafts known to exist in field at deploy time).
- **Keyboard hint surfaces — three locations.** `keyboard-help-overlay.tsx`: VARIANT CELL section now reads `0-9 → Type qty (units)`, `+ = → Add one box (12 / 6 / 4 / 1 units, per pack)`, `- _ → Subtract one box (floor 0)`. Bottom hint bar (the kbd-chip strip below the catalog grid) gained a `+/− box` chip between the `↓↑←→ nav` chip and the `0-9 qty` chip. Critical detail: the hint bar exists in **three separate render paths** — `family-nav-with-tabs.tsx` non-paginated branch (≤17 bases), `sub-product-direct.tsx` non-paginated branch (search → direct sub-product hand-off), and `variant-grid.tsx::PaginationFooter` (>17 bases). All three were patched.
- **Mailto byte-identical contract upheld.** Verified live in production smoke tests:
  - BLACK 50ML=12 / 100ML=24 / 200ML=36 → `UNIVERSAL STAINER BLACK 50ML*12, 100ML*24, 200ML*36`. Volume 10.2 L.
  - CEMENT WB 1L=6 / 4L=4 / 10L=50 → `CEMENT WB 1L*6, 4L*4, 10L*50`. Volume 522 L.
  - Both match what pre-flip would have produced for equivalent inputs.

After Phase 3.4-fix-1 the corner was free; left-edge placement interrupted the left-to-right read between base colour name and typed digit. Moved to right edge in Phase 3.5-fix.

Swapped for `[.group:focus-within_&]:opacity-100` arbitrary-variant form in Phase 3.5-fix. Works on all Tailwind versions; compiles to plain `.group:focus-within .target { opacity: 1 }`.
- **`10L · box 1` / `20L · box 1` column headers** (initial Phase 3.4 behaviour using `packStep` directly). Misleading for drums. Replaced with `packContainerLabel()` helper that returns `"drum"` for 10L/20L/30L and `"bag"` for 40KG in Phase 3.4-fix-2. The two maps stay decoupled so math (`packStep`) and UI text (`packContainerLabel`) can evolve independently.
- **`PACK_STEP_MAP[10L] = 2`** (was the pre-flip value). Wrong for this depot — 10L is a drum, not a box of 2. Fixed to 1 alongside the container label work in Phase 3.4-fix-2.
- **Phase 3.7b chip patched only in `PaginationFooter`** (initial Phase 3.7 implementation — based on CLAUDE_UI.md §51's framing of the hint footer as a single thing). PRIMER → WOOD has 13 bases (non-paginated) and showed no `+/− box` chip after deploy. Investigation revealed three render paths; chip added to `family-nav-with-tabs.tsx` and `sub-product-direct.tsx` non-paginated footers in addition to `PaginationFooter`.

## KEY LEARNINGS FROM THIS SESSION

- **Audit > intuition. Always.** Claude Code's Phase 1 audit caught two material bugs the original session brief missed: the cart panel volume total formula (10.2 L → 122.4 L bug) and the last-order-recall units→boxes division (silent data-loss in recall). Both were single-line fixes once spotted; both would have shipped broken without the audit. The prompt's "do not write code yet" gate at audit time paid for itself many times over.
- **Visual verification is irreplaceable.** Three phases (3.4 cell display, 3.5 +/− buttons, 3.7 hint bar) needed iteration after browser eyeballing — none of these issues would have been caught by `tsc` or by code review alone. The "N box" hint in the cell, the button size + position, and the missing hint chip in non-paginated render paths all surfaced only when the actual rendered output was reviewed in the dev server. Future sessions touching visual elements should keep `dev` running with HMR and gate every visual phase on real-browser confirmation.
- **One-time wipe is fine when no drafts exist in the field.** The draft persistence decision (Option B — wipe old format) was the right call. Migration would have added 15-25 lines of code that runs once then becomes dead weight. Version-bump-key (Option C, my original recommendation) and wipe (Option B) are functionally identical when no drafts are in flight; wipe is simpler.
- **The MAIL_ORDERS §19 description of "the hint footer" as a single thing was misleading.** Three components carry inline hint-footer JSX. The docs framed it as one component with a paginated variant. Next time CLAUDE_UI.md describes a "footer" or "header," be explicit about how many implementation sites carry the same JSX pattern. Consolidation cycle should sweep §19 to clarify this.
- **Decoupling maps (`packStep` vs `packContainerLabel`) was worth the small extra effort.** Two lookups, two jobs. Combining them into a single `packMeta()` returning `{step, container}` would have created false coupling — a future change to step values wouldn't have implied a container label change, and vice versa. The audit phase predicted this and the implementation followed through; well worth keeping as a pattern.
- **Direct-to-main with single-user production stays safe.** Same as the May 11 v5 session — Smart Flow is the only active production user, so no PR gate is needed. Day-one depot use (May 13 morning) will be the real test of the units semantics; if operators get tripped up, a follow-up session can refine the keyboard hints or add operator-facing tooltips. Until depot users come online in numbers, ship velocity wins.
- **Browser verification batch size: 4-5 cases per round is the sweet spot.** Cases A-D, E-F-G, H-I-bar broken into three batches kept the cognitive load manageable. Each batch took ~3-5 min in the browser; total smoke time was about 15 min plus the email overlay verification. Asking for all 9 cases at once would have been unfocused.
