Operator Skip — feature mockup

4 surfaces · Skip top job in operator queue → returns to TM pool

Skip is only available on the operator's TOP job (current pill). Reason picker + optional remark. If reason is "Tinter finished", operator must also pick tinter type and one or more out-of-stock colours. Skipped job returns to TM Pending with an amber "RETURNED — SKIPPED" badge and full skip history.

1 · Operator page — Skip button on the top job

Skip lives in the action footer alongside Save TI / Save TI & Start. Only shown when the current pill job is the operator's top assigned job (NOT in-progress, NOT a future job).

Spec notes:

  • Skip button placement: leftmost in the action cluster (secondary action). Red text on white (NOT red bg) — destructive intent without screaming.
  • Visibility rule: Skip button only renders if (a) selected job is the operator's top job in queue, AND (b) job.status === 'assigned' AND startedAt === null. Hidden on in-progress jobs (Pause is the action for that — Phase 4). Hidden on future jobs.
  • API enforces all three checks again — UI is a UX hint, server is the truth.

Spec notes:

  • Conditional fields: Tinter type + colours only appear when reason = "Tinter finished". For other reasons the modal collapses to: reason + remark + info box.
  • Tinter type toggle: matches the TINTER/ACOTONE toggle on the operator's TI form (UI §49) — gray-900 active. Switching the toggle re-renders the colour grid below.
  • Colour grid: uses the exact pigment colours from TINTER_SHADE_COLORS / ACOTONE_SHADE_COLORS already in tint-operator-content.tsx. 7-column grid for TINTER (2 rows: 7 + 6), 7-col for ACOTONE (2 rows: 7 + 7). Multi-select — click to toggle.
  • Validation gates Skip Job button being clickable: reason required; if tinter finished then type required + ≥1 colour required. Remark always optional.
  • Confirm button: bg-gray-900 per UI §13 modal pattern.
  • API call: POST /api/tint/operator/skip with { assignmentId, reason, tinterType?, outOfStockColours?, remark? } per 3a contract.

Spec notes:

  • Kanban card: amber-3px left border (border-l-amber-500). "RETURNED — SKIPPED Nx" pill at the top in amber. Last skip summary in a tinted amber box (operator · reason · type · colour codes). "View full history →" link below opens the history modal.
  • Skip count: "2×" means the order has been skipped 2 times total (over its lifetime, by any operator). Counts persist across reassignments.
  • Table view: small amber "Skipped 2×" badge under the OBD in the OBD column. Stage column shows "Returned" instead of "Pending" so it's distinguishable from never-assigned. Operator column shows "— (returned)" to indicate the previously-assigned operator is no longer linked.
  • Card sort: returned cards sort like other pending cards by sequenceOrder. No special floating to top — spec doesn't ask for it. Can be added later if Chandresh wants.

Spec notes:

  • Order: newest-first. Most recent skip gets an amber left border + "most recent" tag.
  • Per-event fields: Operator name, reason (human label), tinter type (only if TINTER_FINISHED), colours (only if TINTER_FINISHED — chips coloured by pigment top border), remark (em-dash if empty).
  • API: GET /api/tint/manager/orders/[id]/skip-history → events sorted desc by skippedAt (already done in 3a).
  • Permission: Same as TM page access — Admin OR canView on tint_manager (already enforced server-side in 3a).
  • Empty state: if total = 0, show "No skip events for this OBD" in muted italic. (Shouldn't happen since this modal is only reachable from a returned card.)
