Notes

Behaviour annotations

  • 1
    Primary SO auto-flips on conflict. Setting a 2nd SO row to Primary demotes the current Primary to Backup client-side, then sends one PATCH. The DB partial-unique index is the final guardrail.
  • 2
    Auto-contact ✕ shows confirm dialog. Hover tooltip on the ✕ button reads "Delete this auto-contact? It won't come back unless you re-add the SO above." Click opens the modal shown above.
  • 3
    Greyed inputs are read-only. Name + phone on auto-contacts come from sales_officer_master and refresh on every save. To change them, edit the SO master record (Admin → Sales Officers).
  • 4
    Email field stays editable on auto-contacts. Operators often have a site-specific email for an SO. Stage D sync touches only name + phone; email, isPrimary, and any operator-changed contactRoleId are preserved.
  • 5
    Manual contacts unaffected by sync. Contacts with linkedSalesOfficerId = NULL render with a gray avatar, white inputs, and no link badge. Stage D ignores them entirely. Stage E may flip Primary contact off if a Primary SO claims it.

Avatar colour cue: teal = Primary SO · blue = Backup SO · amber = Junior SO · gray = manual contact.

Sort order: auto-contacts first (Primary → Backup → Junior), divider, then manual contacts (operator-chosen order, isPrimary DESC then id ASC).

No JS in this mockup: all dropdowns, buttons, and pills are static. The role pill-group renders all three states (Primary/Backup/Junior) with the active one highlighted; clicking does nothing.
