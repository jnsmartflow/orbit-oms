## 1. What this application is

Orbit OMS is a depot-level order management system for a paint distribution company operating out of Surat, India. It manages the full lifecycle of customer orders from manual XLS import through tinting, support review, dispatch planning, warehouse picking, and vehicle dispatch.

This is an internal business tool — not a public-facing product. Users are depot staff: dispatchers, support agents, tint operators, warehouse supervisors, and pickers. Each role sees a different interface and has different permissions.

Scale: ~25–35 dispatch plans per day, ~100–200 OBDs (orders) per day, single depot in Phase 1.

## 7. Workflow stages

```
order_created
  ↓ (if tint order)
pending_tint_assignment → tinting_in_progress → tinting_done
  ↓ (all orders converge here)
pending_support
  ↓
dispatch_confirmation   ← Support sets dispatch_status = 'dispatch'
  ↓
dispatched
```

**Routing fork at order_created:**
- `order_type = tint` → `pending_tint_assignment`
- `order_type = non_tint` → `pending_support` directly
- Mixed (tint + non-tint lines) → routes as tint. Entire OBD waits.

**Support visibility (v10 change):** Support can see AND act on ALL orders regardless of tint/non-tint status and regardless of where they are in tinting. Routing fork still exists — Support visibility is an additional layer.

**Stage 4 Hold (v10 change):** If Support sets dispatch_status = hold AND order is already on a draft dispatch plan → system writes a row to `dispatch_change_queue` to notify Dispatcher. Dispatcher manually removes the order. No auto-removal.

## 8. Slot assignment logic (v10 — fully config-driven)

**No hardcoded cutoff times anywhere in the codebase.** All slot rules live in:
- `slot_master` — slot definitions
- `delivery_type_slot_config` — per-delivery-type rules

**Slot determination at import time:**
1. Read `delivery_type_slot_config` for the order's delivery type
2. If `slotRuleType = time_based`: evaluate OBD Email Time column from import file against `windowStart`/`windowEnd` in sortOrder sequence. First match wins.
3. If `slotRuleType = default`: assign that slot regardless of time.
4. `isDefault = true` row = fallback when no window matches.

**Current seed rules for Local delivery type:**
| Window | Slot |
|---|---|
| 00:00 – 10:29 | Morning 10:30 |
| 10:30 – 12:29 | Afternoon 12:30 |
| 12:30 – 15:29 | Evening 15:30 |
| 15:30 – 17:59 | Night 18:00 (default — overridable) |
| 18:00 – 23:59 | Next Day Morning 10:30 |

**Upcountry:** Single default slot = Night 18:00. Overridable.

**Slot override:** Both Support and Dispatcher can override any order's slot. No approval required. Override written to `orders.dispatch_slot_override`. Audit trail written to `order_status_logs`.

## 10. SKU structure (v10)

Each SKU row = one unique product + colour combination. Three FK lookup tables:

```
product_category → product_name → sku_master ← base_colour
```

- `grossWeightPerUnit` REMOVED from sku_master. Weight comes from `import_obd_query_summary.weight` at import time.
- `sku_sub_master` REMOVED. Colour variants are separate sku_master rows with different `baseColourId`.
- Use `base_colour.name = 'N/A'` for non-tint SKUs with no colour variant.

## 11. Sales Officer Group pattern

```
sales_officer_group.salesOfficerId → sales_officer_master
delivery_point_master.salesOfficerGroupId → sales_officer_group
```

- Customer belongs to exactly one group.
- To reassign entire group: `UPDATE sales_officer_group SET salesOfficerId = <new> WHERE id = <group>`
- To move one customer: `UPDATE delivery_point_master SET salesOfficerGroupId = <new_group>`

## 12. Customer route/type inheritance

The system uses a two-level inheritance pattern:

1. **Area level** (default): `area_master.deliveryTypeId` and `area_master.primaryRouteId`
2. **Customer level** (override): `delivery_point_master.deliveryTypeOverride` and `delivery_point_master.primaryRouteId`

Application logic: check customer-level override first → fall back to area value if null.

## 13. The 16 dispatch logic rules (Phase 2+)

**Vehicle assignment (L1–L3):**
- L1: ≤900kg + local = Light. 901–2000kg OR upcountry = Medium. >2000kg = Heavy.
- L2: vehicle_id must be set before plan confirmed.
- L3: vehicle locked permanently once loading starts.

**Priority sort (L4) — 6 tiers:**
1. Key Customer / Key Site
2. Urgent priority
3. Aged ≥ aging_priority_days
4. Earliest dispatch_slot_deadline
5. Address group (same ship_to + slot), heaviest first
6. Weight DESC

**Overflow (L5–L8):** See PHASE_1_BUILD_PROMPTS.md for full rules.
**Rolling wave (L9–L12):** Soft lock → hard lock transitions.
**Warehouse execution (L13–L16):** Picker assignment, discrepancy, escalation, tint gate.

## 14. Audit trail rules — non-negotiable

- `tint_logs` — INSERT-ONLY. Every tint status change = new row.
- `order_status_logs` — INSERT-ONLY. Every order field change = new row with field, old_value, new_value, changed_by, timestamp.

Any UPDATE or DELETE on these tables is wrong. Raise an error at the ORM layer.
