## 1. What this application is

Orbit OMS is a depot-level order management system for a paint distribution company operating out of Surat, India. It manages the full lifecycle of customer orders from SAP import through tinting, support review, dispatch planning, warehouse picking, and vehicle dispatch.

This is an internal business tool — not a public-facing product. Users are depot staff: dispatchers, support agents, tint operators, warehouse supervisors, and pickers. Each role sees a different interface and has different permissions.

Scale: ~25–35 dispatch plans per day, ~100–200 OBDs (orders) per day, single depot in Phase 1.

## 5. Workflow stages — the spine of the application

Every order has a `workflow_stage` field that drives routing. The valid transitions are:

```
order_created
  ↓ (if tint order)
pending_tint_assignment → tinting_in_progress → tinting_done
  ↓ (all orders converge here)
pending_support
  ↓
dispatch_confirmation   ← Support sets dispatch_status = 'dispatch' to reach here
  ↓
dispatched
```

**Routing fork at order_created:**
- `order_type = tint` → `workflow_stage = pending_tint_assignment`
- `order_type = non_tint` → `workflow_stage = pending_support`
- Mixed (TINT + NON TINT) → routes as tint. Entire OBD waits for tint completion.

**Dispatch slot — set ONCE at import, never auto-recalculated:**
- Before 10:30 AM → Same Day 10:30
- After 10:30 AM, before 2:00 PM → Same Day 2:00 PM
- After 2:00 PM → Next Day

## 6. system_config — 9 keys, read from DB always

**Never hardcode these values anywhere in the application. Always read from system_config table.**

| Key | Default | Purpose |
|---|---|---|
| `dispatch_cutoff_time` | 10:30 | Daily cutoff for same-day slot |
| `soft_lock_minutes_before_cutoff` | 30 | When draft → soft lock |
| `hard_lock_minutes_before_cutoff` | 15 | When soft lock → hard lock |
| `ready_escalation_minutes` | 10 | Escalation timer after material ready |
| `upgrade_small_overflow_pct` | 12 | Max overflow % before upgrade suggestion |
| `upgrade_max_dealer_combo` | 3 | Max dealers in concentration check |
| `aging_priority_days` | 2 | Days before order gets tier-3 priority |
| `aging_alert_days` | 3 | Days before escalation alert |
| `change_queue_urgent_alert` | true | Urgent orders bypass change queue |

## 7. The 16 dispatch logic rules — summary

These are implemented in Phases 2 and 3. Keep them in mind when writing any order or planning logic.

**Vehicle assignment (L1–L3):**
- L1: ≤900kg + local = Light. 901–2000kg OR any upcountry = Medium. >2000kg = Heavy. Upcountry always minimum Medium regardless of weight.
- L2: vehicle_id must be set before plan can be confirmed. Modal blocks if null.
- L3: vehicle locked permanently once loading starts. vehicle_locked = TRUE.

**Priority sort (L4) — 6 tiers in order:**
1. Key Customer / Key Site
2. Urgent priority
3. Aged ≥ aging_priority_days
4. Earliest dispatch_slot_deadline
5. Address group (same ship_to + slot), heaviest group first
6. Weight DESC

**Overflow (L5–L8):**
- L5: overflow ≤12% → bump card, dispatcher approves
- L6: overflow >12% AND 2–3 dealer combo ≥900kg → suggest split (Trigger A)
- L7: any combo ≥900kg even before overflow → early warning (Trigger B)
- L8: on split, NEVER inherit original vehicle category. Each resulting vehicle re-derives independently.

**Rolling wave (L9–L12):**
- L9: draft tier + capacity + no disruption → silent auto-slot
- L10: soft lock → dispatcher approval queue with safe_to_add flag
- L11: hard lock → auto-queue to next slot, passive notification
- L12: time transitions at -30min (soft lock) and -15min (hard lock)

**Warehouse execution (L13–L16):**
- L13: picker screen blank until supervisor assigns by OBD
- L14: discrepancy blocks that OBD only. Dispatcher: dispatch short OR hold OBD.
- L15: escalation timer if dispatcher hasn't acted within ready_escalation_minutes
- L16: plan confirm blocked if any tint OBD has tinting_status ≠ tinting_done

## 8. Audit trail rules — non-negotiable

Two tables are INSERT-ONLY and must never be updated or deleted:

- `tint_logs` — every tint status change writes a new row
- `order_status_logs` — every order field change writes a new row with field, old_value, new_value, changed_by, timestamp

Any code that attempts UPDATE or DELETE on these tables is wrong. Raise an error at the ORM layer.
